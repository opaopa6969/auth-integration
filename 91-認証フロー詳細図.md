# 91 — 認証フロー詳細図 (phase/state 別 Mermaid)

[90-起動方法と品質調査メモ.md](90-起動方法と品質調査メモ.md) で見つけた品質問題を、
**「どの変数がどうなってると、どこで何がコケる/出ない」**のレベルで図にしたもの。

1枚にすると長大すぎるので **phase 単位で分割**。各図は **始点 (●START)** と **終点 (■END)** を持ち、
終点には「→ 次は Phase X の始点へ」を書いてある。下の遷移マップが全体のつなぎ。

## phase 遷移マップ

```mermaid
flowchart LR
    A["Phase A: ログイン画面 描画"] -->|メール入力| B["Phase B: Magic Link 送信"]
    A -->|パスキー| D["Phase D: パスキーでログイン"]
    A -->|provider| F["Phase F: 外部 IdP (Part 3)"]
    B --> C["Phase C: Magic Link 検証 = ログイン成立"]
    C -.ログイン済みで.-> E["Phase E: パスキー登録"]
    E -.登録済みなら.-> D
    D --> C2["ログイン成立 /console/"]
    style A fill:#e0e7ff
    style C fill:#dcfce7
    style C2 fill:#dcfce7
    style F fill:#fde68a
```

要点だけ先に:
- **Google ボタンが出ない** = Phase A で `providers` が empty (Part 2 は設計どおり)。
- **パスキーが使えない** = Phase D/E で `WEBAUTHN_RP_ORIGIN` が実 origin と不一致 → finish で検証失敗。
- **ブラウザだけで進めない** = 旧 Phase A に Magic Link フォームが無く Phase B に入れなかった (修正済)。

---

## Phase A — ログイン画面の描画 (`GET /login` → `login.jte`)

「Google ボタンが出るか」「Magic Link 欄が出るか」が決まる phase。

```mermaid
flowchart TD
    START(["●START: ブラウザが GET /login"]) --> H["AuthFlowHandler.loginPage"]
    H --> P["providers = oidcService.enabledProviders()"]
    P --> Q1{"volta-config に idp: の中身がある?<br/>hasIdpSection = !idp.isEmpty"}
    Q1 -->|"idp: [] なので false"| ENV["ENV 検出に fallback"]
    Q1 -->|"idp: に項目あり"| LIST["config の順で enable"]
    ENV --> Q2{"env に GOOGLE_CLIENT_ID 等がある?"}
    Q2 -->|"Part 2 は無し"| EMPTY["providers = empty"]
    Q2 -->|"あれば"| HASP["providers に追加"]
    LIST --> HASP
    EMPTY --> R1["@if(!providers.isEmpty) が false<br/>→ provider 区切り線とボタンを出さない"]
    HASP --> R2["provider ボタンを描画"]
    R1 --> FORM["Magic Link フォーム + パスキーボタンを描画"]
    R2 --> FORM
    FORM --> ENDA["■END: ログイン画面表示<br/>Google ボタンは providers empty なら出ない = 設計どおり<br/>修正前は empty でも区切り線だけ残り画面が切れて見えた<br/>→ 次はユーザの選択で Phase B / D / F の始点へ"]
    style START fill:#e0e7ff
    style EMPTY fill:#fee2e2
    style ENDA fill:#dcfce7
```

> ポイント: `idp: []` は「IdP を明示的に空にした」つもりでも `hasIdpSection()` が **false** に
> なって ENV 検出に落ちる (`OidcService.java:124`)。どっちにせよ Part 2 は env も空なので結果は同じ。

## Phase B — Magic Link 送信 (`POST /auth/magic-link/send`)

```mermaid
flowchart TD
    START(["●START: メール入力 → ログインリンクを送る<br/>fetch POST /auth/magic-link/send"]) --> C{"CSRF チェック: Origin ヘッダは?"}
    C -->|"同一オリジン Origin=http://localhost:28888"| OK1["isAllowedOrigin = true (host=localhost)<br/>→ CSRF exempt"]
    C -->|"Origin 無し / 別オリジン"| NG1["CSRF_INVALID 403"]
    OK1 --> V{"email が valid? (@ を含む)"}
    V -->|"no"| NG2["BAD_REQUEST 400"]
    V -->|"yes"| MK["store.createMagicLink(email, TTL 10分)<br/>enqueue notification"]
    MK --> DEV{"DEV_MODE?"}
    DEV -->|"true"| LINK["response に token / link を含める"]
    DEV -->|"false 本番"| MAIL["メール送信のみ (mailpit)"]
    LINK --> ENDB["■END: 画面に踏めるリンクを表示<br/>→ リンクを踏む = Phase C の始点へ"]
    MAIL --> ENDB2["■END: mailpit / 受信箱にメール<br/>→ リンクを踏む = Phase C の始点へ"]
    style START fill:#e0e7ff
    style NG1 fill:#fee2e2
    style ENDB fill:#dcfce7
    style ENDB2 fill:#dcfce7
```

> ポイント: 同一オリジン POST だから `Origin` が自動で付く → CSRF exempt
> (13章で curl に `-H 'Origin: ...'` を付けていたのと同じ理屈, `Main.java:255`)。

## Phase C — Magic Link 検証 = ログイン成立 (`GET /auth/magic-link/verify?token`)

```mermaid
flowchart TD
    START(["●START: リンクを踏む<br/>GET /auth/magic-link/verify?token=xxx"]) --> T{"token は有効?<br/>used_at 未消費 かつ TTL 内?"}
    T -->|"無効 / 期限切れ"| NG["エラー → /login へ戻る = Phase A の始点"]
    T -->|"有効"| CONSUME["token を consume (used_at セット)"]
    CONSUME --> U{"user は存在する?"}

    U -->|"既存"| EXIST["既存 user を使う"]
    NEW --> SESS["Set-Cookie __volta_session (8h)"]
    EXIST --> SESS
    SESS --> ENDC["■END: ログイン成立 → 302 /console/<br/>X-Volta-User-Id / Tenant-Id / Roles が gateway 経由で todo に流れる<br/>→ ここから Phase E (パスキー登録) に進める"]
    style START fill:#e0e7ff
    style ENDC fill:#dcfce7
    style NEW fill:#fef9c3
```

## Phase D — パスキーでログイン (`POST /auth/passkey/start` → `finish`)

「パスキーが使えない」の主因 (origin 不一致) がここで出る。

```mermaid
flowchart TD
    START(["●START: 🔑 パスキーでログイン"]) --> PRE{"この rpId に登録済みパスキーがある?"}
    PRE -->|"無い"| NOPK["OS ダイアログに候補が出ない<br/>→ 先に Phase E で登録が必要"]
    PRE -->|"ある"| S["POST /auth/passkey/start → challenge 発行"]
    S --> GET["navigator.credentials.get()<br/>OS が Face ID / PIN で本人確認 → 署名"]
    GET --> FIN["POST /auth/passkey/finish"]
    FIN --> ORG{"webauthn4j 検証<br/>ServerProperty.origin == clientDataJSON.origin?"}
    ORG -->|"WEBAUTHN_RP_ORIGIN 未設定 = 既定 :7070<br/>≠ 実 :28888"| FAIL["★旧バグ: origin 不一致で検証失敗<br/>→ ログインできない"]
    ORG -->|"修正後 = :28888 で一致"| OKV["署名検証 OK / sign_count 更新"]
    FAIL --> BACK["/login に戻る = Phase A の始点"]
    OKV --> ENDD["■END: Set-Cookie → /console/ = ログイン成立"]
    style START fill:#e0e7ff
    style FAIL fill:#fee2e2
    style NOPK fill:#fef9c3
    style ENDD fill:#dcfce7
```

## Phase E — パスキー登録 (`/settings/security`, 要ログイン)

「パスキーの種類が少ない (OS ダイアログの選択肢)」がここの `type` で決まる。
そして登録 finish でも **同じ origin 一致が必須**。

```mermaid
flowchart TD
    START(["●START: ログイン済みで /settings/security<br/>+ パスキーを追加"]) --> TYPE{"どの type を選んだ?"}
    TYPE -->|"この端末 type=platform"| P1["attachment=platform / UV=required<br/>→ OS ダイアログは内蔵認証器のみ"]
    TYPE -->|"セキュリティキー type=cross-platform"| P2["attachment=cross-platform / UV=required<br/>→ USB/NFC キー + PIN"]
    TYPE -->|"おまかせ type=any"| P3["attachment 無指定 / residentKey=preferred<br/>→ 別端末(スマホ) やセキュリティキーも候補に出る"]
    P1 --> START2["register/start: challenge +<br/>pubKeyCredParams=[EdDSA -8, ES256 -7, RS256 -257]"]
    P2 --> START2
    P3 --> START2
    START2 --> CREATE["navigator.credentials.create()<br/>OS が鍵ペア生成 (秘密鍵は端末内)"]
    CREATE --> FIN["register/finish"]
    FIN --> ORG{"webauthn4j 検証: ServerProperty.origin が一致?"}
    ORG -->|"WEBAUTHN_RP_ORIGIN 不一致 (:7070)"| FAIL["★旧バグ: 登録 finish が失敗<br/>→ パスキーが1件も保存されない<br/>→ Phase D で永遠に使えない"]
    ORG -->|"一致 (:28888)"| SAVE["store.createPasskey で保存"]
    FAIL --> BACK["エラー表示のまま"]
    SAVE --> ENDE["■END: 登録完了<br/>→ これで Phase D (パスキーでログイン) の始点が使えるようになる"]
    style START fill:#e0e7ff
    style FAIL fill:#fee2e2
    style ENDE fill:#dcfce7
```

> 「OS の選択肢が少ない」と感じたら **type=おまかせ(any)** を使う。platform 固定だと内蔵認証器しか出ない。

## Phase F — 外部 IdP / Google (Part 3 で実装)

```mermaid
flowchart TD
    START(["●START: provider ボタン押下<br/>GET /login?start=1&provider=GOOGLE"]) --> CHK{"provider が enabled?<br/>= Phase A で providers に入ってる?"}
    CHK -->|"Part 2: providers empty → ボタン自体が無い"| NONE["■END: Part 2 では到達不可<br/>→ Phase A の説明どおり (設計)"]
    CHK -->|"Part 3: GOOGLE_CLIENT_ID 設定済み"| GO["state / nonce / PKCE 生成<br/>→ Google 認可画面へ redirect"]
    GO --> CB["GET /callback?code=...<br/>code → id_token 検証 → OidcIdentity"]
    CB --> ENDF["■END: user upsert → Set-Cookie = ログイン成立<br/>(Phase C と同じ合流点)"]
    style START fill:#fde68a
    style NONE fill:#fee2e2
    style ENDF fill:#dcfce7
```

> Google を出すには Part 3 で GCP の OAuth クライアントを作り、env に
> `GOOGLE_CLIENT_ID` 等を入れる (→ 章 [24](24-GCP-OAuth作成.md)/[25](25-volta-auth-proxy起動.md))。
> `GOOGLE_CLIENT_ID` が非空になると `isGoogleEnabled=true` → Phase A の `providers` に GOOGLE が入る。
